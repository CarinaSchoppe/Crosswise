Projektform [Bearbeiten]

Nutze das herausgegebene JavaFX-Projekt mit seinen Einstellungen. Nutze die notwendigen Bibliotheken (JUnit, gson,..) in den dort eingestellten Versionen.

Verwendest Du eine andere Entwicklungsumgebung, so musst Du das Projekt zum Einchecken in die Form des heraugegebenen Projektes bringen. Nur in solchen Projekten werden wir Hilfestellung leisten und bewerten.

 

Programmaufteilung [Bearbeiten]

Trennung GUI-Logik

Es ist darauf zu achten, dass zwischen Oberfläche und Programmlogik sauber getrennt wird. Dafür sollen auf oberster Ebene zwei Pakete "gui" und "logic" angelegt werden. Die Trennung von GUI und Logik ermöglicht ein Redesign des Erscheinungsbilds (welches eventuell von anderen Projektteilnehmern umgesetzt wird), ohne Änderungen in der Logik vornehmen zu müssen. So sollten z. B. andere Bilder, andere Komponenten (die vielleicht ein dreidimensional wirkendes Bild visualisieren) oder sogar andere Grafikbibliotheken (Swing oder etwas ganz Neues) eingesetzt werden können, ohne etwas an Objekten der Spiellogik zu ändern. Bilder sind also in einem Unterpaket von "resources/gui" unterzubringen. Zum Erscheinungsbild gehört auch die Sprache, in der dem Benutzer etwas mitgeteilt wird. Textuelle Ausgaben sind also unbedingt im Grafikpaket unterzubringen.

Im UserInterfaceController sollen nur Routinen zu finden sein, die etwas mit der direkten Oberflächensteuerung zu tun haben. Alle anderen Routinen sind in entsprechende andere Klassen auszulagern. Daher ist in diesem Projekt vorgeschrieben, GUIConnector, JavaFXGUI und FakeGUI zu implementieren.

Als erster Ansatz ist es sinnvoll, eine Klasse JavaFXGUI zu erstellen, die im Konstruktor alle Komponenten erhält, die sie aktualisieren können soll (z.B. diverse Bilder für Karten, Label oder auch ganze Panes). Der UserInterfaceController erzeugt also eine Instanz von JavaFXGUI und gibt sie an die Spiellogik. Die Methoden der JavaFXGUI ermöglichen der Logik, die Ausgabe GUI zu ändern (z.B. Methode showCards(Player player1) zeigt die Karten eines Spielers an, indem die entsprechenden ImageViews aktualisiert werden). Die Klasse mit der Spiellogik erhält im Konstruktor die JavaFXGUI, so dass deren Methoden ausgeführt werden können.

Zum Testen ist es sinnvoll, keine grafische Oberfläche aufbauen zu müssen. Dafür bietet es sich an, ein Interface GUIConnector mit allen für die JavaFXGUI notwendigen Methoden zu gestalten und dieses von der Klasse JavaFXGUI implementieren zu lassen. Außerdem soll das Interface von einer Klasse FakeGUI implementiert werden, die in allen publizierten Methoden nichts ausführt. In der Spiellogik muss jetzt statt der konkreten JavaFXGUI an allen Stellen das Interface GUIConnector genannt werden. So kann die Spiellogik, die eine FakeGUI übergeben bekommt, alle Anzeigemethoden ausführen, ohne dass grafische Komponenten vorhanden sein müssen.


Der UserInterfaceController erzeugt also eine Instanz von JavaFXGUI, um sie bei der Instanziierung der Logik zu übergeben, die Testumgebung erzeugt eine Instanz von FakeGUI, mit der die Logik erzeugt und getestet werden kann. Die Spiellogik erwartet lediglich eine Klasse, die den GUIConnector implementiert.

class GameLogic {
  GUIConnector gui;
  int score;
 
  public GameLogic(GUIConnector gui) { 
    this.gui = gui;
  }
 
  public void setScore(int score) {
    this.score = score;
    gui.showScore(score);
  }
}
 
 interface GUIConnector {
  //enthält kein JavaFX!
  void showScore(int score);
 }

 

class FakeGUI implements GUIConnector {
  void showScore(int score) {}
}

 

 

class JavaFXGUI implements GUIConnector {
  Label this.lblScore;
  JavaFXGUI(Label lblScore) {
    this.lblScore = lblScore;
  }
  void showScore(int score) {
    lblScore.setText(String.valueOf(score));
  }
}

Der GUIConnector gehört in das Logik-Paket, die JavaFXGUI in das GUI-Paket und die FakeGUI in das Paket der Tests. Aus dem Logik-Paket darf ausschließlich das Interface GUIConnector genutzt werden.

Klare Hierachie und Einteilung

Der UserInterfaceController erzeugt eine Logik und eine JavaFXGUI. Er kann also sowohl der Logik Informationen geben, als auch direkt oder über die JavaFXGUI auf der Oberfläche etwas anzeigen. Der UserInterfaceController selbst soll nicht den GUIConnector implementieren.
 
Der UserInterfaceController informiert die Logik, wenn der Nutzer etwas an der Situation ändert. Für die Logik ist es irrelevant, ob z.B. die Karten einer Hand in einem 3*4-Raster angezeigt werden und ob der Nutzer eine Karte mit der linken oder der rechten Maustaste angeklickt hat. Die Logik erfährt also lediglich, dass der Nutzer z.B. die 5. Karte seiner Hand ablegen möchte.,

Die Logik wird an ihrem Zustand etwas ändern (im Beispiel dem aktuellen Spieler die entsprechende Karte abnehmen und auf dem Stapel ablegen). Über den GUIConnector kann sie die Oberfläche aktualisieren (z.B. Hand neu anzeigen, Stapel neu anzeigen). Nun kann die Logik Folgeaktionen durchführen (z.B. prüfen, ob ein Spieler gewonnen hat, oder den nächsten Spieler zum aktuellen machen) und dies über den GUIConnector auf der Oberfläche kenntlich machen.

Der GUIConnector erhält nur logische Informationen, aus denen die entsprechende Anzeige generiert wird. Im Fall einer Fehlermeldung kann hier ein Aufzählungwert übergeben werden, der in der JavaFXGUI zu einem Ausgabetext übersetzt wird. Auf keinen Fall ist der Ausgabestring direkt zu übergeben.  
Der GUIConnector ist lediglich für eine Richtung der Informationsweitergabe verantwortlich. Die Methoden sollten hier also keine Rückgabewerte haben.


Eine JavaFXGUI oder auch der UserInterfaceController darf Elemente der Logik (z.B. Enums, Konstanten, statische Klassen) nutzen.  
Es widerspricht aber einer klaren Trennung, wenn eine JavaFXGUI eine Instanz der Logik vorhält. Benötigt eine Methode der JavaFXGUI Informationen über den Zustand der Logik, so sind diese Informationen (und möglichst keine anderen) als Parameter in die Methode zu geben. Wird also beispielsweise die Belegung einer Zelle benötigt, muss diese übergeben werden, nicht aber das Spielfeld und erst recht nicht die gesamte Logik. Keine Regel ohne Ausnahme: Soll erst nach Beenden einer Animation etwas Logisches passieren - beispielsweise der nächste Spieler am Zug sein, so darf die Logik in die Animationsmethode gereicht werden, damit diese die Methode aus der Logik aufrufen kann.

Kurzcheck

 

Die Trennung von Logik und Oberfläche ist im Groben korrekt, wenn

  • keine Klasse im Logik-Paket eine Klasse aus dem GUI-Paket importiert oder nutzt,
  • im Logik-Paket keine Ausgabetexte erstellt werden, die bei einer Umstellung auf eine andere Sprache übersetzt werden müssten, im GUIConnector also keine Texte zur Ausgabe übergeben werden,
  • keine Methode des GUIConnectors einen Rückgabewert besitzt,
  • nirgendwo Klassen aus "awt" oder "swing" benutzt oder importiert werden.
 

Vorgehensweise und Vorgaben [Bearbeiten]

 

Freiheiten in der Aufgabenstellung

Die Aufgabenstellung bietet nicht in allen Punkten eine genaue Vorgabe, was wie zu lösen ist. An diesen Stellen sind eigene Überlegungen und Entscheidungen erforderlich (und gewünscht), um das Erstellen von Problemlösungen zu fördern. Bei größeren Projekten ist immer wieder die Entscheidung gefordert, wie detailliert man ein Problem bearbeitet (Perfektion der Ausführung versus Zeitaufwand für diese Perfektion) und wie weit man die Effizienz der Lösung optimiert.

Diese Überlegungen (Problemanalyse) und die daraus entstehende Realisation sollten sofort (nicht erst nach Programmfertigstellung) in einer Dokumentation notiert werden, um hinterher die Begründung für die Entscheidung nachvollziehbar darstellen zu können. Dabei soll erkennbar werden, zwischen welchen Varianten eine Abwägung stattgefunden hat.

 

GUI

Lass die GUI so aussehen, wie Du es von einem Standardprogramm erwartest!

  • Gehe davon aus, dass Dein Programm auch auf einfachen Monitoren mit einer Bildschirmauflösung von 1280x1024 benutzt werden soll. Die Mindest- und/oder bevorzugte Größe des Fensters darf natürlich auch kleiner sein.
  • Wenn nicht in der Aufgabenstellung explizit ausgeschlossen, müssen sich die Komponenten bei einer Vergrößerung/Verkleinerung des Fensters mit ihrer Größe anpassen.
  • Ermögliche ein Verändern der Fenstergröße entsprechend der Ziehbewegung (halte also ein Fenster nicht quadratisch oder anders proportional).
  • Verwende z.B.
    • bei sich ausschließenden Auswahlmöglichkeiten RadioButtons,
    • bei an-/auszuschaltenden Optionen Checkboxen,
    • bei Werten, die innerhalb eines Bereichs einzustellen sind, Schieberegler (Slider).
  • Nicht erwartungskonform sind beispielsweise Obermenüpunkte ohne Unterpunkte, eine ComboBox als Untermenüpunkt, Buttons als Ein-/Ausschalter (hiermit sollte immer irgendeine Aktion ausgelöst werden).
  • Nutze für ein Grid die Gridlines nur zum Zweck des Debuggens.
  • Beim Ausführen soll die Warnung WARNUNG: Loading FXML document with JavaFX API of version 17 by JavaFX runtime of version 11.0.2 nicht erscheinen. Der SceneBuilder schreibt seine JavaFX-Version in der FXML-Datei in die Eigenschaften des umgebenden Containers. Diese Version soll in die Version der eingebundenen JavaFX-Libraries geändert werden, um zu verdeutlichen, dass das Programm mit den eingebundenen Libraries keine Probleme aufwerfen wird. Leider überschreibt der SceneBuilder diesen Wert bei jeder Änderung an der FXML, so dass jedes Mal ein händisches Korrigieren notwendig ist.

Vermeide, dass der Benutzer zu viele Klicks machen muss. Mehrere Einstellungen oder Wahlmöglichkeiten gehören also in ein Fenster und nicht in mehrere, die hintereinander aufploppen. Auch zu viele Informationsfenster (Alerts) nerven mit der Zeit. Arbeite hierfür lieber mit einer Statuszeile.

 

Logik

 

Es dürfen alle Klassen aus den per default installierten Packages der in RZ1-5 installierten JavaFX-Installation genutzt werden. Die Verwendung von Collections ist einer eigens programmierten Liste vorzuziehen.

Beim Programmieren sind die Richtlinien zu beachten, die auch bei der Übung galten (Programmierstil, Kommentierung, Zugriffe über Getter/Setter).

Lege die Logik universell an, so dass Spiele mit weniger Spielern/Spielsteinen/Karten/Feldern gespielt werden können. Der Standardkonstruktor sollte hierfür einen Konstruktor mit Parametern aufrufen und die Parameter aus privaten Konstanten entnehmen. So kannst Du überschaubare Spiele testen. Die GUI wird bei Erzeugen der Logik dieser die gewünschten Größen/Anzahlen mitgeben, sie muss also nicht variabler sein, als die Spielregeln verlangen.
 

Man kann beim Design der Klassenstruktur ganz gut von der Sprache ausgehen: "Ein Spiel hat ein Spielfeld." → Das Spielfeld ist ein Attribut des Spiels. Evtl. lohnt es sich, das Spielfeld in eine eigene Klasse auszulagern und ihm verschiedene Fähigkeiten (Methoden) zu geben, damit die Spiel-Klasse nicht so umfangreich wird.
"Die schlaue KI ist eine spezielle KI." → Hier bietet sich meist Vererbung an.

  • Es ist überaus sinnvoll, für eine Position eine eigene Klasse zu erstellen. Diese kann zumindest validieren, dass keine Positionen mit negativen Werten erstellt werden können. Sie kann Methoden zum Ermitteln aller Nachbarpositionen anbieten, oder auch für den Vergleich (Interface Comparable implementieren) und dadurch eine einfache Listensortierung ermöglichen.
  • Vermutlich möchtest Du bei der Entwicklung Debugausgaben auf der Konsole ausgeben. Diese für die Abgabe alle zu entfernen, ist sehr mühsam. Nutzt Du für Debugausgaben eine eigens geschriebene Methode zur Ausgabe eines Strings, die die Ausgabe nur im Falle einer mit TRUE belegten Konstante ausführt, so kannst Du diese Konstante vor der Abgabe/Erstellung der jar einfach auf FALSE setzen.
  • Versuche die Verwendung von Klassenvariablen so weit wie möglich einzuschränken. Setze keine Klassenvariablen ein, die Du aus anderen Zuständen (z.B. Arraylänge) ermitteln kannst. Und versuche, wenn Du Klassenvariablen deklariert hast, so wenig wie möglich aus anderen Klassen auf diese Variablen zuzugreifen (auch nicht über Getter), sondern sie stattdessen an die entsprechenden Prozeduren und Funktionen als Parameter zu übergeben.
  • Gib den Methoden Rückgabewerte statt sie void zu deklarieren und Klassenvariablen setzen zu lassen. Du vermeidest dadurch mögliche Inkonsistenzen durch nicht sauber gepflegte Klassenvariablen, sparst den Platz für die Klassenvariablen und kannst die Methode einfach testen. Möchtest Du mehrere Daten zurückgeben, hilft Dir evtl. ein Record.
  • Schränke jede Sichtbarkeit so weit wie möglich ein (Attribute immer private). Getter werden nur public angeboten, wenn unbedingt notwendig. Public Getter dürfen keine Referenzen auf die Attribute herausgeben, sondern lediglich Kopien davon, um mögliche Manipulationen von aussen zu verhindern. Setter werden nur public angeboten, wenn das hiermit gesetzte Attribut nach Erstellung des Objekts verändert werden können muss.
  • Setze keine magic numbers ein, sondern verwende Konstanten!
  • Strings sollten keine logische Information enthalten sondern ausschließlich Text. Soll der Text für den Benutzer ausgegeben werden, so ist er erst in der GUI zu erstellen.
  • Erstelle für die Spiellogik einen Konstruktor, der eine gesamte Spielsituation in für den Menschen gut lesbarer Form aufnehmen kann. So kann beispielsweise bei einem Aufruf von 
    new TicTacToeGame("_ x _" +
                      "o _ x" +
                      "o _ _")
    sofort die Spielsituation erfasst werden und bereits getestet werden, ob ein Spieler gewonnen hat. Der übergebene String muss lediglich auf das Spielfeld übersetzt werden.
  • Erstelle mehrere Konstruktoren, die sich gegenseitig aufrufen. Gibt es für das TicTacToe-Spiel noch einen Konstruktor, der außer dem Spielfeldstring den aktuellen Spieler annimmt, so kann beispielsweise der optimale Zug ermittelt werden.
  • Entferne ungenutzten automatisch generierten Code und Kommentar.
  • Setze als Variablentyp einen möglichst allgemeinen ein (z.B. GUIConnector statt JavaFXGUI oder auch List statt ArrayList).
  • Mögliche Inkonsistenzen sollen vermieden werden. Ein Attribut darf also nicht in verschiedenen Formen gedoppelt gehalten werden, sondern ist bei Bedarf umzuformen. Die Konstante für eine Information (z.B. Spielfeldgröße) darf nur an einer Stelle vorliegen (also nicht in GUI und Logik). Hier muss entweder das Grid dynamisch aus der in der Logik festgelegten Spielfeldgröße aufgebaut werden oder die Logik muss die Information von der GUI erhalten, in welchem Format das Spielfeld aufzubauen ist.
  • Wird ein Log erstellt, so schließe die Datei nach jedem Schreibvorgang, damit bei einem unerwarteten Programmabbruch die vorausgegangenen Schritte nachvollzogen werden können.
  • Entschließt Du Dich für einen Aspekt im Klassendesign, so notiere Dir die Überlegungen, die Du angestellt und verworfen hast, um später in der Realisationsanalyse der Dokumentation verschiedene Lösungsmöglichkeiten diskutieren zu können.
  • Füge aussagekräftige Modulköpfe und Inline-Dokumentation ein, dies wirkt sich auf die Bewertung der Dokumentation aus.

Tests

  • Nutze keine automatische Testgenerierung, da hiermit stumpf jede Methode einen Test generiert bekommt, statt dass die relevanten Methoden bzw. Situationen sinnvoll getestet werden.
  • Getter und Setter brauchen nicht getestet werden, auch wenn bei Settern ein Test für nicht erlaubte Werte sinnvoll wäre.
  • Teste stattdessen Spielsituationen: Generiere eine Spielsituation (es kann nur für diesen Zweck ein package-private Konstruktor geschrieben werden, der eine konkrete Spielsituation mitten im Spiel aufnimmt), führe einen Zug aus und prüfe dann, ob die Spielsituation mit Deiner Erwartung übereinstimmt (ist der Spieler wie erwartet am Zug, sieht das Spielbrett wie erwartet aus). Für einen "normalen" Zug und für jeden Zug, der zu einer Sonderregelung führt, muss mindestens ein Test existieren.
  • Entsteht die Testklasse in einem gleichnamigen Paket wie die zu testende Klasse, kann in ihr auf alle im selben Paket sichtbaren Elemente zugegriffen werden.
  • Verwende für den Vergleich von Arrays assertArrayEquals(..). assertEquals(int[], int[]) vergleicht lediglich die Referenzen, assertTrue(Arrays.deepEquals(int[][], int[][])) gibt keine Angabe der Fehlerposition. assertArrayEquals(int[][], int[][]) führt einen tiefen Vergleich aus!
  • Füge keine Ausgaben in Deine Tests, nur fehlschlagende Tests sollten Ausgaben erzeugen. Jedes JUnit-Assert kann als ersten Parameter einen String erhalten, der im Fehlerfall ausgegeben wird. Dieser String kann bei Bedarf umfangreiche Informationen enthalten (z.B. ganze Spielfelder).
  • Denke daran, auch nach der Abnahme des Zwischenstands zahlreiche JUnit-Tests zu erstellen. Diese wirken sich stark auf die Bewertung der Programmierersicht aus.
